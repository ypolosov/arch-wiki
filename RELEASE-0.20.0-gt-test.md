# arch-wiki v0.20.0 — gt-тест (mirror hierarchy fix: корень = arc42 only)

> Поверх v0.19.0. Исправление иерархии зеркала. Содержимое страниц НЕ меняется (body байт-идентичен);
> меняется только РОДИТЕЛЬ 15 страниц → при перепубликации они переезжают в правильную секцию.

## Проблема
Под корнем «Архитектура решения — граф знаний» (`index.md`) висели 27 детей = 12 arc42-хабов (верно)
+ **15 лишних** (13 concepts + `glossary.md` + `utility-tree.md`). Верхнеуровневое оглавление засорялось.

## Что изменилось
- **`parentSourceOf`** (ConfluenceTree): корень теперь родитель **только arc42 §1–§12**. Не-нумерованные
  страницы вешаются под секцию по умолчанию: concept → §4, entity → §5, `glossary` → §12,
  `utility-tree` → §10. Переопределяется per-page через frontmatter **`arc42_parent: <NN>`**.
- Обратная совместимость: сигнатура расширена опциональным `arc42Hubs` (без него — прежнее поведение).

## Acceptance (проверено локально на gt)
- `render-confluence --all`: дети `index.md` = **12, все arc42, 0 лишних** (было 15).
- 15 переехали: 13 concepts → §4, glossary → §12, utility-tree → §10.
- **body-diff = 0** (контент байт-идентичен); hash-diff = 15 ровно у переехавших (их contentHash
  включает позицию → пере-опубликуются и сдвинутся; остальные 141 байт-стабильны).
- `verify-mirror` = ok, 156, 0 нарушений. Все тесты: 342 + 17 e2e зелёные.

## Как применить на gt
```
claude plugin marketplace update && claude plugin update arch-wiki@ypolosov-marketplace --scope local
# РЕСТАРТ. arch-wiki version → 0.20.0
arch-wiki render-confluence --all    # проверь: у index.md только 12 arc42-детей
# затем /arch-wiki:publish (2 прохода) — 15 страниц ПЕРЕЕДУТ под §4/§10/§12;
#   остальные без дрейфа. Это egress в Confluence — в рабочее пространство SD.
```
(Опц.: у конкретной страницы задай `arc42_parent: 08` во frontmatter, чтобы повесить её под другую секцию.)

## Откат
Плагин — на v0.19.0 (`ref`). В Confluence — перепубликация вернёт прежние родители.
